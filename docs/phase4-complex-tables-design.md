# Phase 4: Complex Stateful Tables — Design Document

## Overview
This document covers the 7 remaining Cloud tables that are **not yet implemented in CAPS**. These are complex, stateful tables where ownership (Cloud vs CAPS) must be carefully defined before implementation. No code changes should be made until ownership and conflict resolution strategies are agreed upon.

## Tables Covered

### 1. Tip Pooling (2 tables)

#### `tip_allocations`
- **Cloud Schema**: employee_id, check_id, amount, tip_type, business_date, created_by, etc.
- **Ownership Question**: Where are tips allocated? Currently Cloud-only via EMC.
- **Recommended Ownership**: **Cloud-owned, sync DOWN to CAPS** (read-only at POS).
- **Rationale**: Tip allocation involves complex business rules (percentage splits, job-code weighting) that are better managed centrally. POS only needs to display allocations.
- **CAPS Role**: Display tip info for employees; no writes.

#### `tip_pool_runs`
- **Cloud Schema**: property_id, business_date, pool_policy_id, status, total_amount, etc.
- **Ownership**: **Cloud-owned** — EMC runs tip pool calculations.
- **CAPS Role**: None. This is an EMC/manager function.

### 2. Inventory (2 tables)

#### `inventory_stock`
- **Cloud Schema**: property_id, inventory_item_id, current_quantity, last_count_date, etc.
- **Ownership Question**: Does POS decrement stock on sale?
- **Recommended Ownership**: **Dual-write with CAPS authority during offline**.
  - Online: POS writes to CAPS → syncs to Cloud.
  - Offline: CAPS is sole authority; reconcile on reconnect.
- **Conflict Resolution**: Last-write-wins with CAPS offline window taking priority.

#### `inventory_transactions`
- **Cloud Schema**: property_id, item_id, transaction_type (sale, waste, adjustment, count), quantity, etc.
- **Ownership**: **CAPS creates (sale transactions), Cloud creates (adjustments, counts)**.
- **Sync**: Bi-directional with transaction_type as conflict discriminator.

### 3. Timecard Extensions (3 tables)

#### `timecard_edits`
- **Cloud Schema**: timecard_id, field_name, old_value, new_value, edited_by_id, reason, etc.
- **Ownership**: **Cloud-owned** — EMC managers edit timecards.
- **CAPS Role**: Receive edited timecards via sync; display edit history read-only.
- **Complication**: CAPS creates timecards (clock in/out), but edits come from Cloud. Need timecard version field.

#### `timecard_exceptions`
- **Cloud Schema**: timecard_id, exception_type (missing_punch, overtime_threshold, etc.), severity, resolved, etc.
- **Ownership**: **Cloud-owned** — generated by Cloud's payroll engine.
- **CAPS Role**: Display exceptions. CAPS cannot resolve them.

#### `time_off_requests`
- **Cloud Schema**: employee_id, start_date, end_date, request_type, status, approved_by_id, etc.
- **Ownership**: **Cloud-owned** — EMC/scheduling feature.
- **CAPS Role**: Read-only. Display approved time-off on POS scheduling views if implemented.

## Conflict Resolution Patterns

### Pattern A: Cloud-Owned, Sync Down (Read-Only at CAPS)
Used by: tip_allocations, tip_pool_runs, timecard_edits, timecard_exceptions, time_off_requests
- Cloud writes → delta sync pushes to CAPS
- CAPS stores locally for offline display
- No CAPS writes to these tables

### Pattern B: CAPS-Owned, Sync Up
Used by: (already implemented) timecards, terminal_sessions, break_attestations, break_violations
- CAPS creates records → cloud_synced=0 → sync queue pushes to Cloud
- Cloud stores as received; no Cloud edits back to CAPS

### Pattern C: Dual-Write with Offline Authority
Used by: inventory_stock, inventory_transactions
- Both sides can write
- CAPS has authority during offline window
- Reconciliation on reconnect: merge transactions by timestamp, recalculate stock levels
- Requires `last_modified_by` and `source` fields for conflict detection
