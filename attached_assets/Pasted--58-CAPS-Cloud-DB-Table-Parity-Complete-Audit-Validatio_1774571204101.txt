#58 - CAPS ↔ Cloud DB Table Parity — Complete Audit, Validation & Fix
Objective
Close every table-sync gap between Cloud PostgreSQL and CAPS SQLite, add runtime validation to system diagnostics so gaps are caught automatically on every update, and establish the rule that new EMC features are always synced down but NOT enabled until explicitly activated.

Context
Cloud DB has 139 tables; CAPS has 88 (including 9 CAPS-only infra tables)
Config-sync currently syncs 55 config tables from cloud → CAPS
21 cloud tables are completely absent from CAPS that arguably should exist
For each missing table, three breaks in the pipeline: cloud doesn't send it, CAPS schema doesn't have it, CAPS config-sync doesn't process it
Rules
NEW-FEATURE SYNC RULE: Every table added to Cloud/EMC MUST be included in the config-sync payload and CAPS schema. New features sync down as data but remain disabled at POS until the relevant option flag/bit is turned on in EMC. This prevents breaking live sites.
No automated tests, no DB queries, no code changes without explicit approval.
All service-host logs use getLogger(...) — no bare console.log.
Part 1: Table Comparison Report + Diagnostics Validation
1A — Add "Cloud ↔ CAPS Table Parity" section to CAPS Diagnostic endpoint
File: service-host/src/routes/api.ts — extend the existing /caps/diagnostic/summary route

Add a new section to the diagnostic summary response that reports:

cloudTablesExpected: hard-coded list of all Cloud tables that should exist in CAPS (config + operational)
capsTablesPresent: actual tables found in the SQLite database (query sqlite_master)
missingFromCaps: tables in cloudTablesExpected but not in SQLite
extraInCaps: tables in SQLite not in cloudTablesExpected (CAPS-only infra tables — should list them but mark as "expected CAPS-only")
cloudOnlyByDesign: tables intentionally NOT synced (admin/EMC infrastructure — e.g., emc_sessions, emc_users, stress_test_results, cal_*, service_host_*, etc.)
parity: boolean — true only if missingFromCaps is empty
syncedTableCounts: for each synced config table, show { table, capsCount, lastSyncAt }
The hard-coded expected-table list serves as a contract. Whenever a new table is added to Cloud schema, the developer MUST add it to either cloudTablesExpected or cloudOnlyByDesign — if they forget, diagnostics will flag it as missing.

Add a new collapsible section called "Cloud ↔ CAPS Table Parity" that shows:
