MASTER TASK PLAN — REBUILD RUNTIME FLOW FOR ELECTRON / WS / KDS / CAPS / CLOUD

We are NOT starting over from scratch.
We are freezing the architecture and correcting the runtime model.

==================================================
1) FINAL ARCHITECTURE MODEL
==================================================

A. Electron App
The Electron app is the terminal shell installed on each workstation or KDS device.
It is responsible for:
- initial installer/setup flow
- saving device identity and selected environment locally
- launching the UI
- talking to CAPS only for runtime operations
- showing truthful connection/device status

Electron is NOT the store authority.
Electron is NOT allowed to decide cloud vs local for live POS/KDS transactions.
Electron is NOT allowed to silently fallback to cloud for live runtime functions.

B. CAPS / Service Host
CAPS is the local site controller and single runtime authority for the location.
CAPS is responsible for:
- local SQLite operational database
- workstation and KDS device registration/state
- employees, roles, privileges used at runtime
- menu, modifiers, layouts, taxes, tenders, discounts
- checks, items, payments, service charges, locks
- KDS tickets and KDS device communication
- local journals and sync queue
- syncing up to cloud
- syncing config down from cloud

C. Cloud
Cloud is upstream only.
Cloud is responsible for:
- configuration master
- enterprise reporting
- device activation/bootstrap data
- receiving synced transactions/journal data from CAPS
- remote management/admin

Cloud is NOT in the live POS/KDS transaction path.

==================================================
2) REQUIRED DEVICE BOOT FLOW
==================================================

When Electron is first installed:

STEP 1 — ENVIRONMENT SETUP
User enters:
- customer cloud URL
- username/password
- property
- device type (WS or KDS)
- specific workstation/KDS device identity

STEP 2 — DEVICE ACTIVATION
Electron authenticates to cloud ONLY for setup/activation.
Cloud returns:
- property id
- workstation or KDS device id
- device name
- CAPS activation configuration
- service host URL if this device is the CAPS server or should connect to CAPS
Electron stores this locally.

STEP 3 — LOCAL DEVICE PROFILE
Electron writes local device profile containing:
- cloud URL
- property id
- device id
- device name
- device type
- serviceHostUrl
- whether this machine is the CAPS host
- last known config version

STEP 4 — CAPS STARTUP DECISION
If device is CAPS host:
- start embedded service-host locally
- wait for CAPS ready state
- then continue boot

If device is not CAPS host:
- do not start embedded service-host
- connect to configured CAPS host over LAN/local network
- wait for CAPS ready state
- then continue boot

STEP 5 — CAPS READINESS CHECK
Electron must not enter normal runtime state until CAPS confirms:
- DB open
- schema ready
- config loaded
- device/workstation records loaded
- websocket server ready
- runtime APIs ready

CAPS must expose a clear readiness endpoint:
GET /health/ready
Returns:
- status: starting | ready | degraded
- dbReady
- configReady
- websocketReady
- deviceRegistryReady

STEP 6 — UI BOOT
Electron opens UI immediately but in boot/loading state.
UI must show:
- loading device configuration
- connecting to store server
- loading menu/layout
- ready

UI must NOT enter broken runtime state if CAPS is not ready.
It must wait or retry cleanly.

==================================================
3) WS RUNTIME MODEL
==================================================

Workstation runtime behavior:

- sign in through CAPS
- load workstation context from CAPS
- load menu/layout/modifiers from CAPS
- create/open/pickup/transfer/send/pay only through CAPS
- all reads/writes come from CAPS
- no cloud fallback

If CAPS is unavailable:
- show store server unavailable
- do not partially load broken UI
- do not attempt live transactions

If check is created on WS01:
- WS02 sees it from CAPS
- KDS sees it from CAPS
- reports see it from CAPS

==================================================
4) KDS RUNTIME MODEL
==================================================

KDS runtime behavior:

- KDS boots using stored device identity
- loads KDS device config from CAPS
- opens local CAPS websocket or polling channel
- receives tickets only from CAPS
- bump/recall/priority actions go only to CAPS

KDS must not require cloud to function once configured.
If cloud is down but CAPS is up:
- KDS still works

If CAPS is unavailable:
- KDS shows waiting for store server
- not a permanent broken config screen
