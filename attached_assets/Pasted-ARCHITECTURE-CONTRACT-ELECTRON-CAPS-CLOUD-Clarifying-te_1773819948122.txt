ARCHITECTURE CONTRACT — ELECTRON / CAPS / CLOUD

Clarifying terminology first:

- Electron app = client shell running on POS terminals and KDS devices
- CAPS / service-host = local store authority and runtime engine
- Cloud = central sync, configuration, and enterprise reporting

For pilot, this is the required operating model:

1) ELECTRON APP (CLIENT SHELL)
The Electron app is the UI used by operators (POS + KDS).

It must:
- send all live POS and KDS operations to CAPS
- render responses from CAPS
- not depend on cloud for live check or KDS data

Even though Electron currently contains routing/mode logic, it must behave as a client of CAPS for all store operations.

---

2) CAPS / SERVICE-HOST (STORE AUTHORITY)
CAPS is the single source of truth for the location.

CAPS must own ALL runtime store data and behavior:
- employees and privileges
- menu, modifiers, pricing
- checks and check items
- discounts and service charges
- tenders and payments
- open checks list and check locks
- pickup, transfer, reopen, split, merge
- KDS ticket creation and status
- local reporting
- transaction journal and sync queue

If a check is created, updated, or paid:
→ it must exist in CAPS first
→ all Electron clients must see the same result from CAPS

---

3) CLOUD (UPSTREAM ONLY)
Cloud is NOT part of the live POS execution path.

Cloud is only responsible for:
- receiving synced transactions from CAPS
- distributing configuration to CAPS
- enterprise/back-office reporting

Cloud must NOT be used as fallback for live POS/KDS operations.

---

NON-NEGOTIABLE RULES

- Electron → CAPS → local SQLite commit → success to UI → background sync to cloud
- Electron must NOT fall through to cloud for any live POS/KDS route
- If CAPS returns 404 or fails for a live POS route, that is a BUG — do not fall back to cloud
- All workstations and KDS devices must read the same data from CAPS
